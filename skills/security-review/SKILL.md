# 보안 리뷰 스킬 (Security Review Skill)

이 스킬은 모든 코드가 보안 최선 관행을 따르도록 하고 잠재적인 취약점을 식별하는 데 도움을 줍니다.

## 활성화 시점

- 인증(Authentication) 또는 권한 부여(Authorization) 구현 시
- 사용자 입력 처리 또는 파일 업로드 기능 구현 시
- 새로운 API 엔드포인트 생성 시
- 시크릿(Secrets) 또는 자격 증명(Credentials) 작업 시
- 결제 관련 기능 구현 시
- 민감한 데이터를 저장하거나 전송할 때
- 외부 제3자 API를 통합할 때

## 보안 체크리스트 (Security Checklist)

### 1. 시크릿 관리 (Secrets Management)
- **절대 금지**: 소스 코드에 API 키, 비밀번호 등을 하드코딩하지 마십시오.
- **권장 사항**: 항상 환경 변수(`process.env`)를 사용하고, 사용 전 존재 여부를 검증하십시오. `.env.local` 등은 반드시 `.gitignore`에 포함시켜야 합니다.

### 2. 입력값 검증 (Input Validation)
- `Zod`와 같은 스키마 검증 라이브러리를 사용하여 모든 사용자 입력을 검증하십시오.
- 파일 업로드 시 파일 크기, 타입(MIME type), 확장자를 엄격히 제한하십시오. 블랙리스트 방식이 아닌 **화이트리스트 방식**을 사용하십시오.

### 3. SQL 인젝션 방지
- SQL 쿼리 작성 시 문자열 결합을 절대 피하십시오.
- 항상 **매개변수화된 쿼리(Parameterized Queries)**를 사용하거나 신뢰할 수 있는 ORM/쿼리 빌더를 활용하십시오.

### 4. 인증 및 권한 부여
- JWT 토큰은 브라우저의 `localStorage`가 아닌, XSS에 안전한 **httpOnly 쿠키**에 저장하십시오.
- 민감한 작업 수행 전에는 반드시 권한 체크를 수행하십시오. Supabase를 사용하는 경우 행 레벨 보안(RLS)을 활성화하십시오.

### 5. XSS 및 CSRF 방지
- 사용자로부터 받은 HTML 콘텐츠는 반드시 `DOMPurify` 등으로 새니타이징(Sanitize)하십시오.
- 콘텐츠 보안 정책(CSP) 헤더를 설정하십시오.
- 상태를 변경하는 작업에는 CSRF 토큰을 사용하고, 모든 쿠키에 `SameSite=Strict` 옵션을 적용하십시오.

### 6. 전송 및 노출 방지
- 로그에 비밀번호, 토큰 등 민감한 정보가 남지 않도록 마스킹 처리하십시오.
- 사용자에게 노출되는 에러 메시지는 구체적인 내부 정보를 담지 않도록 일반적인 메시지만 제공하십시오. 스택 트레이스(Stack trace)를 노출하지 마십시오.

### 7. 보안 테스트 및 배포 전 점검
- 인증 요구, 권한 제한, 입력값 거부, 속도 제한(Rate limiting) 등을 검증하는 자동화된 보안 테스트를 작성하십시오.
- 배포 전에는 시크릿 하드코딩 여부, HTTPS 강제 적용, 보안 헤더 설정, 의존성 취약점 점검(`npm audit`) 등을 반드시 확인하십시오.

**기억하십시오**: 보안은 선택 사항이 아닙니다. 단 하나의 취약점이 전체 플랫폼을 위험에 빠뜨릴 수 있습니다. 확실하지 않을 때는 항상 가장 보수적이고 안전한 방법을 선택하십시오.
