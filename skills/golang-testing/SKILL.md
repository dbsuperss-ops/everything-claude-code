# Go 테스트 패턴 (Go Testing Patterns)

TDD 방법론을 따르며 신뢰할 수 있고 유지보수가 쉬운 테스트 작성을 위한 포괄적인 Go 테스트 패턴을 안내합니다.

## 활성화 시점

- 새로운 Go 함수나 메서드를 작성할 때
- 기존 코드에 테스트 커버리지를 추가할 때
- 성능이 중요한 코드에 대해 벤치마크를 생성할 때
- 입력값 검증을 위해 퍼즈(Fuzz) 테스트를 구현할 때
- Go 프로젝트에서 TDD 워크플로우를 적용할 때

## Go를 위한 TDD 워크플로우 (RED-GREEN-REFACTOR)

1. **RED**: 먼저 실패하는 테스트를 작성합니다.
2. **GREEN**: 테스트를 통과시키기 위한 최소한의 코드를 작성합니다.
3. **REFACTOR**: 테스트 통과 상태를 유지하면서 코드를 개선합니다.

## 테이블 기반 테스트 (Table-Driven Tests)

Go 테스트의 표준 패턴입니다. 최소한의 코드로 다양한 케이스를 포괄할 수 있습니다.
- 익명 구조체 슬라이스를 사용하여 입력값과 기댓값을 정의합니다.
- `t.Run()`을 사용하여 각 케이스를 서브 테스트로 실행합니다.

## 테스트 헬퍼 (Test Helpers)

- `t.Helper()`: 에러 발생 시 헬퍼 함수 내부가 아닌 호출한 위치를 표시합니다.
- `t.Cleanup()`: 테스트가 끝날 때 리소스를 자동으로 정리합니다.
- `t.TempDir()`: 테스트용 임시 디렉토리를 생성하고 자동 삭제합니다.

## 골든 파일 (Golden Files)

`testdata/` 디렉토리에 저장된 예상 출력 파일과 실제 결과를 비교하는 방식입니다. 대량의 출력 결과 검증에 유용합니다.

## 인터페이스를 이용한 모킹 (Mocking)

- 종속성(예: 데이터베이스, 외부 API)을 인터페이스로 정의합니다.
- 테스트에서는 해당 인터페이스를 구현한 Mock 객체를 주입하여 동작을 시뮬레이션합니다.

## 벤치마크 (Benchmarks)

`Benchmark`로 시작하는 함수를 작성하여 코드의 실행 시간 및 메모리 할당량을 측정합니다.
- `b.ResetTimer()`: 설정(Setup) 시간을 제외합니다.
- `b.ReportAllocs()`: 메모리 할당 지표를 포함합니다.

## 퍼즈 테스트 (Fuzzing, Go 1.18+)

무작위 데이터를 입력하여 엣지 케이스나 보안 취약점을 발견합니다. `f.Add()`로 기본 입력을 제공하고 `f.Fuzz()`로 실행합니다.

## 테스트 커버리지

- `go test -cover`: 기본 커버리지 확인
- `go test -coverprofile=coverage.out`: 상세 보고서 생성
- `go tool cover -html=coverage.out`: 브라우저에서 시각적으로 확인

## 최선 관행 (Best Practices)

### 권장 사항 (DO)
- **먼저 테스트를 작성하십시오 (TDD)**.
- **테이블 기반 테스트를 활용하십시오**.
- 구현이 아닌 **코드의 동작을 테스트하십시오**.
- 독립적인 테스트에는 `t.Parallel()`을 사용하십시오.

### 지양 사항 (DON'T)
- 프라이빗 함수를 직접 테스트하지 마십시오 (공개 API를 통해 우회 테스트).
- 동기화를 위해 `time.Sleep()`을 사용하지 마십시오 (채널이나 조건 변수 사용).
- 에러 경로(Error path) 테스트를 소홀히 하지 마십시오.

**기억하십시오**: 테스트는 곧 문서입니다. 코드가 어떻게 사용되어야 하는지 보여주어야 합니다. 명확하게 작성하고 최신 상태를 유지하십시오.
