# C++ 코딩 표준 (C++ Core Guidelines 기반)

[C++ Core Guidelines](https://isocpp.github.io/CppCoreGuidelines/CppCoreGuidelines)에서 파생된 현대적 C++(C++17/20/23)을 위한 포괄적인 코딩 표준입니다. 타입 안전성, 리소스 안전성, 불변성 및 코드의 명확성을 강조합니다.

## 활성화 시점

- 새로운 C++ 코드(클래스, 함수, 템플릿 등) 작성 시
- 기존 C++ 코드 검토 및 리팩토링 시
- C++ 프로젝트의 아키텍처 결정 시
- 코드베이스 전반에 걸쳐 일관된 스타일을 적용할 때
- 언어 기능 선택 시 (예: `enum` vs `enum class`, 생 포인터 vs 스마트 포인터 등)

## 핵심 원칙

가이드라인 전반을 관통하는 6가지 핵심 원칙입니다:

1. **모든 곳에 RAII 적용** (P.8, R.1): 리소스의 수명을 객체의 수명에 바인딩하십시오.
2. **기본적으로 불변성(Immutability) 유지** (P.10, Con.1-5): `const`/`constexpr`로 시작하고, 변경이 필요한 경우만 예외로 둡니다.
3. **타입 안전성 확보** (P.4, I.4): 컴파일 타임에 에러를 방지하기 위해 타입 시스템을 활용하십시오.
4. **의도를 명확히 표현** (P.3, F.1): 이름, 타입, 컨셉(Concepts)은 그 목적을 명확히 전달해야 합니다.
5. **복잡성 최소화** (F.2-3): 단순한 코드가 올바른 코드입니다.
6. **포인터 의미론보다 값 의미론 선호** (C.10, R.3-5): 값에 의한 반환과 스코프 객체 사용을 선호하십시오.

## 인터페이스 및 철학 (P.*, I.*)

- **I.1**: 인터페이스를 명시적으로 만드십시오.
- **I.2**: 비-const 전역 변수를 피하십시오.
- **I.4**: 인터페이스에 정확하고 강력한 타입을 사용하십시오.
- **I.11**: 생 포인터(Raw pointer)나 참조로 소유권을 이전하지 마십시오.

```cpp
// 좋은 예: 불변이며 강력한 타입의 인터페이스
struct Temperature { double kelvin; };
Temperature boil(const Temperature& water);

// 나쁜 예: 소유권과 단위가 불분명함
double boil(double* temp);
```

## 함수 (F.*)

- **F.16**: "입력" 파라미터의 경우, 복사 비용이 저렴하면 값으로, 비싸면 `const&`로 전달하십시오.
- **F.20**: "출력" 값의 경우, 출력 파라미터보다 반환 값을 사용하십시오.
- **F.21**: 여러 값을 반환할 때는 struct를 정의하여 반환하십시오.
- **F.43**: 지역 객체에 대한 포인터나 참조를 절대 반환하지 마십시오.

## 클래스 및 계층 구조 (C.*)

- **C.20 (Rule of Zero)**: 기본 동작(생성자, 소멸자 등)을 컴파일러가 생성하게 할 수 있다면 직접 정의하지 마십시오.
- **C.21 (Rule of Five)**: 복사/이동/소멸자 중 하나라도 직접 정의해야 한다면, 5가지 모두를 적절히 처리하십시오.
- **C.35**: 베이스 클래스의 소멸자는 public virtual이거나 protected non-virtual이어야 합니다.
- **C.46**: 인자가 하나인 생성자는 `explicit`으로 선언하십시오.
- **C.128**: 가상 함수에는 `virtual`, `override`, `final` 중 정확히 하나만 명시하십시오.

## 리소스 관리 (R.*)

- **R.1**: RAII를 사용하여 리소스를 자동으로 관리하십시오.
- **R.3**: 생 포인터(`T*`)는 소유권이 없는 "관찰자"여야 합니다.
- **R.11**: `new`와 `delete`를 직접 호출하지 말고 스마트 포인터를 사용하십시오.
- **R.21**: 소유권을 공유해야 하는 경우가 아니면 `shared_ptr`보다 `unique_ptr`을 선호하십시오.

## 식 및 문장 (ES.*)

- **ES.25**: 수정을 의도하지 않는 한 객체를 `const` 또는 `constexpr`로 선언하십시오.
- **ES.47**: `0`이나 `NULL` 대신 `nullptr`을 사용하십시오.
- **ES.48**: C 스타일 캐스트를 피하고 `static_cast` 등을 사용하십시오.
- **ES.50**: `const`를 강제로 제거(cast away)하지 마십시오.

## 에러 처리 (E.*)

- **E.2**: 함수가 할당된 작업을 수행할 수 없을 때는 예외를 던지십시오.
- **E.15**: 값으로 던지고(Throw by value), 참조로 잡으십시오(Catch by reference).
- **E.16**: 소멸자, 메모리 해제, swap 함수는 절대 실패하면 안 됩니다.

## 동시성 및 병렬성 (CP.*)

- **CP.20**: `lock()`/`unlock()`을 직접 호출하지 말고 RAII 방식의 `lock_guard` 등을 사용하십시오.
- **CP.44**: `lock_guard`나 `unique_lock` 객체에 반드시 이름을 명명하십시오.
- **CP.21**: 여러 뮤텍스를 획득할 때는 데드락 방지를 위해 `std::scoped_lock`을 사용하십시오.

## 퀵 체크리스트

작업 완료 전 다음 항목을 확인하십시오:
- [ ] 생 `new`/`delete`가 없는가? (스마트 포인터/RAII 사용)
- [ ] 변수가 기본적으로 `const`/`constexpr`인가?
- [ ] 1인자 생성자에 `explicit`이 붙어 있는가?
- [ ] `enum class`를 사용하였는가?
- [ ] `0`/`NULL` 대신 `nullptr`을 사용하였는가?
- [ ] 베이스 클래스 소멸자가 올바르게 선언되었는가? (virtual 여부)
- [ ] 헤더 파일에 `using namespace`가 전역으로 선언되지 않았는가?
- [ ] 매직 넘버 대신 이름을 가진 상수를 사용하였는가?

**기억하십시오**: 현대적인 C++는 리소스를 직접 관리하는 복잡함을 줄이고 타입 안전성을 강화하는 방향으로 발전하고 있습니다. 가이드라인을 준수하여 더 안전하고 유지보수하기 쉬운 코드를 작성하십시오.
