# Eval Harness 스킬

Claude Code 세션을 위한 공식 평가 프레임워크로, 평가 주도 개발(Evaluation-Driven Development, EDD) 원칙을 구현합니다.

## 적용 시점

* AI 보조 워크플로우를 위한 평가 주도 개발(EDD) 환경을 구축할 때
* Claude Code 작업 완료 여부에 대한 명확한 기준(Pass/Fail)을 정의할 때
* pass@k 지표를 사용하여 에이전트의 신뢰성을 측정할 때
* 프롬프트나 에이전트 로직 변경 시 회귀 테스트(Regression test) 스위트를 생성할 때
* 다양한 모델 버전 간의 에이전트 성능을 벤치마킹할 때

## 철학

평가 주도 개발(EDD)은 평가를 "AI 개발을 위한 유닛 테스트"로 간주합니다:

* 구현 **이전**에 예상 동작을 정의합니다.
* 개발 과정에서 지속적으로 평가를 실행합니다.
* 변경 사항마다 기존 기능의 저하(Regression) 여부를 추적합니다.
* 신뢰성 측정을 위해 pass@k 지표를 사용합니다.

## 평가 유형

### 역량 평가 (Capability Eval)
Claude가 이전에는 수행하지 못했던 작업을 수행할 수 있는지 테스트합니다.

```markdown
[역량 평가: 기능 이름]
작업: Claude가 수행해야 할 작업 설명
성공 기준:
  - [ ] 기준 1
  - [ ] 기준 2
  - [ ] 기준 3
예상 출력: 결과물에 대한 상세 설명
```

### 회귀 평가 (Regression Eval)
변경 사항이 기존 기능을 망가뜨리지 않았는지 확인합니다.

```markdown
[회귀 평가: 기능 이름]
기준점: Git SHA 또는 체크포인트 이름
테스트:
  - 기존 테스트-1: 통과/실패
  - 기존 테스트-2: 통과/실패
결과: X/Y 통과 (이전 결과: Y/Y)
```

## 채점 방식 (Grader Types)

### 1. 코드 기반 채점 (Code-based Grader)
코드를 사용하여 결정론적으로 확인합니다.

```bash
# 파일 내에 특정 패턴이 포함되어 있는지 확인
grep -q "export function handleAuth" src/auth.ts && echo "PASS" || echo "FAIL"

# 테스트 통과 여부 확인
npm test -- --testPathPattern="auth" && echo "PASS" || echo "FAIL"

# 빌드 성공 여구 확인
npm run build && echo "PASS" || echo "FAIL"
```

### 2. 모델 기반 채점 (Model-based Grader)
Claude를 사용하여 주관적이거나 개방형인 출력을 평가합니다.

```markdown
[모델 채점자 프롬프트]
다음 코드 변경 사항을 평가하십시오:
1. 명시된 문제를 해결했는가?
2. 구조가 잘 잡혀 있는가?
3. 에지 케이스(Edge cases)를 처리했는가?
4. 에러 처리가 적절한가?

점수: 1-5 (1=매우 불량, 5=매우 우수)
추론: [이유 설명]
```

### 3. 인간 채점 (Human Grader)
수동 검토가 필요한 항목으로 표시합니다.

```markdown
[인간 검토 필요]
변경 사항: 수정된 내용 설명
사유: 수동 검토가 필요한 이유
위험 수준: 낮음/중간/높음
```

## 지표 (Metrics)

### pass@k
"k번의 시도 중 최소 한 번 성공할 확률"

* **pass@1**: 첫 번째 시도 성공률
* **pass@3**: 3번의 시도 내 성공률
* **권장 목표**: pass@3 > 90%

### pass^k
"k번의 모든 시도에서 성공할 확률"

* 높은 신뢰성이 요구되는 지표
* **pass^3**: 연속 3번 성공
* 주요 핵심 로직(Critical paths)에 적용

## 평가 워크플로우

### 1. 정의 (코딩 시작 전)
```markdown
## 평가 정의: 기능-xyz

### 역량 평가
1. 새로운 사용자 계정 생성 가능
2. 이메일 형식 유효성 검사 수행
3. 비밀번호 보안 해싱 적용

### 회귀 평가
1. 기존 로그인 기능 유지
2. 세션 관리 로직 불변
3. 로그아웃 프로세스 완결성

### 성공 지표
- 역량 평가: pass@3 > 90%
- 회귀 평가: pass^3 = 100%
```

### 2. 구현
정의된 평가를 통과할 수 있도록 코드를 작성합니다.

### 3. 평가 실행
```bash
# 역량 평가 실행 및 PASS/FAIL 기록
# 회귀 테스트 실행 (예: npm test)
```

### 4. 보고서 생성
```markdown
평가 보고서: 기능-xyz
========================
역량 평가: 3/3 통과
회귀 평가: 3/3 통과

지표:
  pass@1: 67% (2/3)
  pass@3: 100% (3/3)

상태: 검토 가능 (Ready for Review)
```

## 통합 패턴

### 구현 시작 전
```
/eval define 기능-이름
```
`.claude/evals/기능-이름.md`에 평가 정의 파일을 생성합니다.

### 구현 도중
```
/eval check 기능-이름
```
현재 상태에서 평가를 실행하고 상태를 보고합니다.

### 구현 완료 후
```
/eval report 기능-이름
```
최종 평가 보고서를 생성합니다.

## 평가 저장소 구조

프로젝트 내부에 평가 항목을 저장하십시오:
```
.claude/
  evals/
    feature-xyz.md      # 평가 정의
    feature-xyz.log     # 실행 이력
    baseline.json       # 회귀 기준점 데이터
```

## 베스트 프랙티스

1. **코딩 전 평가 정의**: 성공 기준을 명확히 함으로써 사고를 체계화합니다.
2. **자주 평가 실행**: 회귀 문제를 조기에 발견합니다.
3. **지속적인 pass@k 추적**: 신뢰성 추세를 모니터링합니다.
4. **코드 채점 우선**: 확률적인 모델 채점보다 결정론적인 코드 채점을 선호하십시오.
5. **보안 항목은 인간 검토 병행**: 보안 검사를 완전히 자동화하지 마십시오.
6. **빠른 평가 속도 유지**: 평가가 느리면 아무도 실행하지 않게 됩니다.
7. **평가 파일도 코드처럼 관리**: 평가는 코드와 동등한 수준의 산출물입니다.

## 제품 수준 평가 (v1.8)

단위 테스트만으로 동작의 품질을 충분히 판단하기 어려울 때 제품 평가(Product Eval)를 사용하십시오.

### pass@k 가이드
* `pass@1`: 즉각적인 신뢰도
* `pass@3`: 재시도 허용 시 실제 운영 신뢰도
* `pass^3`: 안정성 테스트 (3번 모두 성공 필수)

권장 임계값:
* 역량 기반: pass@3 >= 0.90
* 회귀 기반: 배포 필수 경로의 경우 pass^3 = 1.00

### 피해야 할 안티 패턴
* 알려진 평가 데이터에만 맞도록 프롬프트를 과적합(Overfitting)시키는 행위
* 정상 경로(Happy path)만 측정하는 행위
* 성공률에만 집착하여 비용과 대기 시간(Latency) 증가를 간과하는 행위
* 배포 관문에서 불안정한 채점기(Flaky grader)를 허용하는 행위
