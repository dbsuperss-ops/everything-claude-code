# 테스트 주도 개발(TDD) 워크플로우

모든 코드 개발이 TDD 원칙을 따르고 포괄적인 테스트 커버리지를 갖추도록 보장하는 스킬입니다.

## 적용 시점

* 새로운 기능이나 모듈을 작성할 때
* 버그나 이슈를 수정할 때
* 기존 코드를 리팩토링할 때
* 새로운 API 엔드포인트를 추가할 때
* 신규 UI 컴포넌트를 생성할 때

## 핵심 원칙

### 1. 테스트 우선 (Test First)
항상 테스트를 먼저 작성하고, 그 테스트를 통과시키기 위한 최소한의 코드를 구현하십시오.

### 2. 커버리지 요구 사항
* **최소 80% 커버리지 유지** (단위 + 통합 + E2E 합산)
* 모든 에지 케이스(Edge cases) 포함
* 에러 시나리오 및 경계 조건(Boundary conditions) 테스트

### 3. 테스트 유형
* **단위 테스트(Unit)**: 개별 함수, 유틸리티, 컴포넌트 로직 (순수 함수 중심)
* **통합 테스트(Integration)**: API 엔드포인트, DB 작업, 서비스 간 상호작용
* **종단간 테스트(E2E)**: Playwright 등을 이용한 실제 사용자 흐름 및 UI 상호작용

## TDD 워크플로우 단계

1. **사용자 여정(User Journey) 정의**: "사용자로서 나는 [동작]을 원하며, 이를 통해 [이득]을 얻는다" 형식으로 정의.
2. **테스트 케이스 생성**: 정의된 여정에 대한 포괄적인 테스트 코드를 작성합니다.
3. **테스트 실행 (실패 확인)**: 아직 구현 전이므로 테스트는 실패해야 합니다 (`Red`).
4. **코드 구현**: 테스트를 통과할 수 있는 가장 단순한 코드를 작성합니다 (`Green`).
5. **테스트 재실행**: 작성한 코드가 테스트를 통과하는지 확인합니다.
6. **리팩토링**: 테스트 성공을 유지하면서 코드 중복 제거, 명명 개선, 성능 최적화를 수행합니다 (`Refactor`).
7. **커버리지 검증**: `npm run test:coverage` 등을 실행하여 80% 이상인지 확인합니다.

## 테스트 파일 구조 예시

```
src/
├── components/          # 컴포넌트와 동일 위치에 .test.tsx 배치
│   └── Button/
│       ├── Button.tsx
│       └── Button.test.tsx
├── app/api/             # API 라우트와 동일 위치에 .test.ts 배치
│   └── markets/
│       ├── route.ts
│       └── route.test.ts
└── e2e/                 # 별도 디렉터리에 시나리오별 .spec.ts 배치
    └── auth.spec.ts
```

## 피해야 할 실수 (Anti-patterns)

* **구현 상세 테스트**: 내부 상태나 변수값을 직접 체크하지 말고, 사용자가 보는 결과물을 테스트하십시오.
* **취약한 선택자**: CSS 클래스명(`.css-xyz`) 대신 의미 있는 선택자(`[data-testid="submit-button"]`)를 사용하십시오.
* **테스트 간 의존성**: 각 테스트는 독립적이어야 합니다. 이전 테스트 결과에 의존하지 않도록 개별 테스트마다 데이터를 셋업하십시오.

**핵심**: 테스트는 선택이 아닌 필수입니다. 테스트는 자신감 있는 리팩토링, 신속한 개발, 그리고 운영 환경의 안정성을 보장하는 가장 강력한 안전망입니다.
