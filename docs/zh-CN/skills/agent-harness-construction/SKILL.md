# 에이전트 하네스(Agent Harness) 구축

에이전트의 계획 수립, 도구 호출, 오류 복구 및 완료 상태 수렴 방식을 개선하고자 할 때 이 스킬을 사용하십시오.

## 핵심 모델

에이전트의 출력 품질은 다음 요소들에 의해 결정됩니다:

1. 액션 공간(Action Space)의 품질
2. 관찰(Observation)의 품질
3. 복구(Recovery)의 품질
4. 컨텍스트 예산(Context Budget)의 품질

## 액션 공간 설계

1. 안정적이고 명확한 도구 이름을 사용하십시오.
2. 입력 스키마는 가급적 좁은 범위를 우선으로 정의하십시오.
3. 결정론적(Deterministic)인 출력 형태를 반환하십시오.
4. 격리가 불가능한 경우가 아니라면 '만능 도구' 사용을 피하십시오.

## 도구의 세분화 규칙 (Granularity Rules)

* 리스크가 큰 작업(배포, 마이그레이션, 권한 설정 등)에는 마이크로 도구를 사용하십시오.
* 일반적인 수정/읽기/검색 루프에는 중간 크기의 도구를 사용하십시오.
* 네트워크 왕복 오버헤드가 주요 비용인 경우에만 매크로 도구를 사용하십시오.

## 관찰(Observation) 설계

모든 도구의 응답에는 다음 항목이 포함되어야 합니다:

* `status`: success|warning|error
* `summary`: 한 줄로 요약된 결과
* `next_actions`: 실행 가능한 다음 단계 명시
* `artifacts`: 파일 경로 또는 ID

## 에러 복구 규약 (Error Recovery Contract)

각 에러 경로에는 다음 내용이 포함되어야 합니다:

* 근본 원인 힌트 (Root Cause Hint)
* 안전한 재시도 지침
* 명확한 중단 조건

## 컨텍스트 예산 관리

1. 시스템 프롬프트는 최소한으로 유지하고 변경하지 마십시오.
2. 방대한 가이드 정보는 필요할 때만 로드되는 '스킬'로 옮기십시오.
3. 긴 문서를 직접 포함하기보다는 파일을 참조하도록 구성하십시오.
4. 단순히 토큰 임계값에 도달했을 때가 아니라, 작업 단계의 경계에서 압축을 수행하십시오.

## 아키텍처 패턴 가이드

* **ReAct**: 경로가 불확실한 탐색적 작업에 가장 적합합니다.
* **함수 호출 (Function Calling)**: 구조화되고 결정론적인 프로세스에 적합합니다.
* **혼합 모드 (권장)**: ReAct를 통한 계획 수립 + 타입화된 도구를 통한 실행.

## 벤치마킹 (Benchmarking)

다음 항목들을 추적하십시오:

* 완료율 (Completion Rate)
* 작업당 재시도 횟수
* pass@1 및 pass@3 지표
* 성공한 작업당 비용

## 안티 패턴 (Anti-patterns)

* 의미가 겹치는 도구가 너무 많은 경우.
* 복구 힌트가 없는 불투명한 도구 출력.
* 후속 단계 안내 없이 에러만 출력하는 경우.
* 관련 없는 참조 자료로 인한 컨텍스트 오버로드.
