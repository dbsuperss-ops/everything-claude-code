# /learn-eval - 추출, 평가 및 저장

기존 `/learn` 명령어를 확장하여, 세션에서 유용한 지식을 추출한 후 품질 게이트(Quality Gate)를 거쳐 최적의 위치에 저장하도록 관리합니다.

## 추출 대상 (무엇을 배우는가?)

다음과 같은 가치 있는 정보를 찾습니다:

1. **에러 해결 패턴**: 근본 원인(Root Cause) + 해결 방법 + 재사용 가능성
2. **디버깅 기법**: 일반적이지 않은 분석 단계, 효과적인 도구 조합
3. **우회 방법 (Workarounds)**: 라이브러리의 특이점, API 제한 사항, 버전별 수정 사항
4. **프로젝트 특화 패턴**: 팀내 컨벤션, 아키텍처 결정 사항, 통합 패턴

## 처리 프로세스

1. 현재 세션을 검토하여 추출 가능한 패턴을 탐색합니다.
2. 가장 가치 있고 재사용 가능한 인사이트를 식별합니다.
3. **저장 위치 결정**:
   * 질문: "이 패턴이 다른 프로젝트에서도 유용할까요?"
   * **전역 (Global)** (`~/.claude/skills/learned/`): 2개 이상의 프로젝트에서 공통으로 쓰일 기술 (Bash 호환성, LLM API 특성, 범용 디버깅 등)
   * **프로젝트 (Project)** (현재 프로젝트 내 `.claude/skills/learned/`): 해당 프로젝트 전용 지식 (특정 설정값의 특이성, 프로젝트 전용 아키텍처 등)
   * 불확실한 경우 **전역**을 선택합니다 (전역에서 프로젝트로 옮기는 것이 반대보다 쉽습니다).
4. 다음 형식으로 스킬 파일 초안을 작성합니다:
   ```markdown
   ---
   name: 패턴명
   description: "130자 이내의 짧은 설명"
   user-invocable: false
   origin: auto-extracted
   ---
   # [설명적인 패턴 이름]
   **추출일:** [날짜]
   **문맥:** [이 패턴이 적용되는 상황 요약]

   ## 문제 상황
   [해결하고자 하는 구체적인 문제 설명]

   ## 해결 방법
   [기술적 해결책 및 코드 예시]

   ## 적용 시점
   [트리거 조건]
   ```
5. **저장 전 자체 평가 (Self-Evaluation)**: 다음 기준에 따라 1~5점을 부여합니다.
   * **구체성 (Concreteness)**: 코드 예시가 풍부한가?
   * **실행 가능성 (Actionability)**: 즉시 따라 할 수 있는 단계인가?
   * **범위 적합성 (Scope Fit)**: 이름, 트리거, 내용이 일관적인가?
   * **독창성 (Non-Redundancy)**: 기존 스킬과 중복되지 않는가?
   * **커버리지 (Coverage)**: 주요 사례와 예외 상황을 모두 다루는가?
   * *모든 항목에서 3점 이상을 받을 때까지 초안을 수정합니다.*
6. **사용자 확인**: 저장 경로, 평가표, 초안 최종안을 보여주고 승인을 받습니다.
7. 지정된 위치에 저장합니다.

## 평가 기준 상세 (Scoring Rubric)

| 항목 | 1점 | 3점 | 5점 |
|-----------|---|---|---|
| 구체성 | 추상적 원칙만 있음 | 대표적 코드 예시 포함 | 다양한 사용 사례와 풍부한 예시 |
| 실행 가능성 | 무엇을 할지 불분명함 | 주요 단계를 이해할 수 있음 | 즉시 실행 가능하며 엣지 케이스 포함 |
| 범위 적합성 | 너무 넓거나 좁음 | 대체로 적절함 | 패턴과 내용이 완벽하게 일치함 |
| 독창성 | 기존 스킬과 거의 같음 | 일부 중복되나 시각이 다름 | 완전히 새로운 가치 제공 |
| 커버리지 | 대상 작업의 일부만 다룸 | 주요 케이스 포함 | 변이형과 함정까지 모두 포함 |

**핵심**: Learn-Eval은 단순히 정보를 저장하는 것을 넘어, 미래의 나(또는 에이전트)가 즉시 활용할 수 있는 고품질의 지식 자산을 구축하는 과정입니다.
