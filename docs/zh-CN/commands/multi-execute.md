# /execute - 다중 모델 협업 실행 (Multi-Model Execution)

수립된 계획으로부터 프로토타입을 확보하고, Claude가 이를 프로덕션급 코드로 리팩토링하여 구현한 뒤, 다중 모델 감사를 거쳐 최종 인도하는 과정을 관리합니다.

## 핵심 프로토콜

* **언어 규정**: 도구/모델과의 상호작용은 **영어**로, 사용자와의 소통은 사용자의 언어(**한국어**)로 수행합니다.
* **코드 주권**: 외부 모델(Codex, Gemini)은 **파일 시스템 쓰기 권한이 없습니다.** 모든 수정은 Claude가 직접 수행합니다.
* **프로토타입 리팩토링**: 외부 모델의 출력물(Unified Diff)은 '가공되지 않은 프로토타입'으로 간주하며, 반드시 프로덕션급 코드로 리팩토링해야 합니다.
* **단계별 검증**: 현재 단계의 결과물이 검증되기 전까지는 다음 단계로 넘어가지 않습니다.
* **전제 조건**: `/ccg:plan` 출력에 대해 사용자가 명시적으로 "Y"라고 답한 경우에만 실행합니다.

---

## 다중 모델 호출 사양

**호출 구문 (병철 실행 시 `run_in_background: true` 사용)**:
```bash
# 세션 재개 호출 (권장) - 구현 프로토타입 확보
~/.claude/bin/codeagent-wrapper --backend <codex|gemini> resume <SESSION_ID> - "$PWD" <<'EOF'
ROLE_FILE: <프롬프트 경로>
<TASK>
요구사항: <작업 설명>
컨텍스트: <계획 내용 및 대상 파일 정보>
</TASK>
OUTPUT: Unified Diff Patch 형식으로만 출력. 실제 수정은 금지함.
EOF
```

**모델별 역할 분담**:
| 단계 | Codex (백엔드) | Gemini (프론트엔드) |
|-------|-------|--------|
| **구현** | 로직, 알고리즘, API 설계 전문 | UI, 컴포넌트, 스타일링 전문 |
| **리뷰** | 보안, 성능, 로직 정확성 감사 | 접근성, 디자인 일관성, UX 감사 |

## 실행 워크플로우

### [단계 0: 계획 읽기 (Prepare)]
1. **입력 식별**: 계획 파일 경로(`.claude/plan/xxx.md`) 또는 직접적인 작업 설명을 확인합니다.
2. **내용 파싱**: 작업 유형, 단계별 절차, 핵심 파일 목록, `SESSION_ID`를 추출합니다.
3. **사전 확인**: 필수 정보가 누락되었거나 사용자의 실행 승인("Y")이 확인되지 않은 경우 다시 확인을 요청합니다.
4. **작업 라우팅**:
   * **프론트엔드**(UI, 스타일): Gemini 배정
   * **백엔드**(API, DB, 로직): Codex 배정
   * **풀스택**: 두 모델 병렬 배정

### [단계 1: 신속한 컨텍스트 수집 (Retrieval)]
계획에 명시된 '핵심 파일' 목록을 기반으로 세밀한 코드 검색을 수행합니다.
* 입구 포인트 파일, 의존성 모듈, 관련 타입 정의 등을 수집합니다.
* 구현에 필요한 충분한 문맥이 확보되었는지 확인합니다.

### [단계 2: 프로토타입 확보 (Prototype)]
라우팅된 모델을 통해 구현 초안을 확보합니다.
* **Gemini**: 디자인 및 시각적 기반이 되는 CSS/React/Vue 프로토타입을 생성합니다. (백엔드 로직 제안은 무시 가능)
* **Codex**: 논리적 추론 능력을 바탕으로 백엔드 로직 및 알고리즘 초안을 생성합니다.
* 결과물은 반드시 **Unified Diff Patch** 형식이어야 합니다.

### [단계 3: 코드 구현 (Implement)]
Claude가 주도권을 가지고 다음 작업을 수행합니다:
1. **패치 분석**: 외부 모델이 제안한 변경 사항을 면밀히 검토합니다.
2. **로직 가상 실행**: 변경 사항을 적용했을 때의 일관성과 부작용(Side Effects)을 미리 시뮬레이션합니다.
3. **리팩토링 및 정리**: 프로토타입을 **가독성 높고 유지보수가 용이한 기업용 코드**로 재작성합니다.
   * 불필요한 주석을 제거하고 자가 설명적(Self-explanatory) 코드를 지향합니다.
4. **최소 범위 적용**: 요구사항에 명시된 범위에만 국한하여 수정하며, 기존 기능에 영향을 주지 않도록 주의합니다.
5. **자체 검증**: 린트(Lint), 타입 체크, 단위 테스트를 실행하여 무결성을 확인합니다.

### [단계 4: 감사 및 인도 (Audit & Delivery)]
1. **교차 감사**: 구현 완료 후 즉시 Codex와 Gemini를 병렬 호출하여 코드 리뷰를 수행합니다.
   * **Codex**: 보안, 성능, 에러 처리, 로직 정확성 중심
   * **Gemini**: 디자인 일관성, 사용자 경험(UX) 중심
2. **통합 및 수정**: 두 모델의 피드백을 수렴하여(백엔드는 Codex 우선, 프론트엔드는 Gemini 우선) 필요한 경우 최종 수정을 가합니다.
3. **인도 보고**: 변경 요약, 감사 결과 및 이후 권장되는 테스트 단계를 포함하여 사용자에게 보고합니다.

## 핵심 규칙 요약

1. **에이전트 주권**: 모든 파일 수정은 오직 Claude(본인)만이 수행하며 외부 모델은 쓰기 권한이 없습니다.
2. **품질 우선**: 프로토타입은 반드시 리팩토링 과정을 거쳐야 합니다.
3. **신뢰 기반**: 백엔드는 Codex, 프론트엔드는 Gemini의 의견을 우선시합니다.
4. **안전성**: 모든 변경 사항은 다중 모델 교차 감사를 통과해야 합니다.
