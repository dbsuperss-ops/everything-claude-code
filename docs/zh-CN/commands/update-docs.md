# 문서 업데이트 (Update-Docs) 명령어

코드의 변경 사항을 문서에 즉시 반영하여, 개발자가 항상 최신화된 가이드를 참고할 수 있도록 관리합니다.

## 1단계: 단일 진실 공급원(SSOT) 식별

시스템은 다음 파일들을 기반으로 문서 내용을 추출합니다:

| 원본 파일 | 생성 대상 문서 |
|--------|-----------|
| `package.json` 스크립트 | 사용 가능한 명령어 가이드 |
| `.env.example` | 환경 변수 설정 문서 |
| `openapi.yaml` / 라우트 파일 | API 엔드포인트 레퍼런스 |
| 소스 코드 Export 구문 | 공개 API(Public API) 문서 |
| `Dockerfile` / Docker Compose | 인프라 설정 및 구동 가이드 |

## 2단계: 스크립트 레퍼런스 생성

1. `package.json` (또는 `Makefile`, `Cargo.toml`, `pyproject.toml`)을 분석합니다.
2. 모든 실행 가능 스크립트와 그에 대한 설명을 추출합니다.
3. 다음과 같은 참조 표를 생성합니다:

```markdown
| 명령어 | 설명 |
|---------|-------------|
| `npm run dev` | 핫 리로드가 포함된 개발 서버 기동 |
| `npm run build` | 타입 체크를 포함한 프로덕션 빌드 실행 |
| `npm test` | 커버리지 측정을 포함한 테스트 수트 실행 |
```

## 3단계: 환경 변수 문서화

1. `.env.example` 등 템플릿 파일을 읽습니다.
2. 각 변수의 용도와 필수 여부를 분류합니다.
3. 예상되는 형식과 유효한 값의 범위를 기록합니다.

```markdown
| 변수명 | 필수 | 설명 | 예시 |
|----------|----------|-------------|---------|
| `DATABASE_URL` | 예 | PostgreSQL 연결 문자열 | `postgres://user:pass@host:5432/db` |
| `LOG_LEVEL` | 아니오 | 로그 상세도 (기본: info) | `debug`, `info`, `warn`, `error` |
```

## 4단계: 핵심 가이드 업데이트 (CONTRIBUTING, RUNBOOK)

* **CONTRIBUTING.md**: 개발 환경 설정, 테스트 작성법, 코드 스타일 준수 사항, PR 체크리스트 갱신
* **RUNBOOK.md**: 배포 절차, 헬스 체크 엔드포인트, 장애 대응(Troubleshooting) 시나리오, 롤백 절차 보완

## 5단계: 문서 시효성 점검

1. 90일 이상 수정되지 않은 문서 파일을 탐색합니다.
2. 최근의 소스 코드 변경 이력과 교차 검증합니다.
3. 내용이 유효하지 않을 가능성이 있는 문서를 식별하여 사용자에게 알립니다.

## 문서 관리 규칙

* **자동 생성 원칙**: 수동 편집보다는 코드로부터 자동 추출된 내용을 우선합니다.
* **수동 영역 보존**: 사람이 직접 작성한 고유한 설명 영역은 훼손하지 않고 유지합니다.
* **구역 표시**: 자동 생성된 영역은 `<!-- AUTO-GENERATED -->` 태그를 사용하여 구분합니다.
* **명시적 요청**: 사용자의 명시적인 명령이 있을 때만 새로운 문서 파일을 생성합니다.

**핵심**: Update-Docs 명령어는 코드와 문서 사이의 간극을 줄여, 문서가 낡아 쓸모없어지는 '문서 부패' 현상을 방지하고 팀의 지식 공유 효율을 높입니다.
