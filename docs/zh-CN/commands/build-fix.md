# 빌드 및 오류 수정

최소한의 안전한 변경을 통해 빌드 및 타입 에러를 단계적으로 수정합니다.

## 1단계: 빌드 시스템 감별

프로젝트의 빌드 도구를 식별하고 빌드를 실행합니다:

| 식별자 | 빌드 명령어 |
|-----------|---------------|
| `package.json` 내 `build` 스크립트 | `npm run build` 또는 `pnpm build` |
| `tsconfig.json` (TypeScript 전용) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m py_compile` 또는 `mypy .` |

## 2단계: 에러 분석 및 그룹화

1. 빌드 명령어를 실행하고 stderr(표준 에러)를 캡처합니다.
2. 에러를 파일 경로별로 그룹화합니다.
3. 의존성 순서에 따라 정렬합니다 (가져오기/타입 에러를 먼저 수정하고 로직 에러를 나중에 수정).
4. 진행 상황 추적을 위해 전체 에러 개수를 산출합니다.

## 3단계: 수정 루프 (에러 당 한 번씩 처리)

각 에러에 대해 다음 과정을 반복합니다:

1. **파일 읽기** — 읽기 도구를 사용하여 에러 문맥(에러 발생 지점 주변 10라인)을 확인합니다.
2. **진단** — 근본 원인(누락된 가져오기, 타입 불일치, 구문 에러 등)을 판단합니다.
3. **최소한의 수정** — 에러 해결을 위한 가장 작은 단위의 코드 변경을 수행합니다.
4. **빌드 재실행** — 에러가 해결되었는지, 새로운 에러가 발생하지 않았는지 확인합니다.
5. **다음 단계** — 남은 에러를 계속 처리합니다.

## 4단계: 가드레일 (중단 기준)

다음 상황에서는 작업을 멈추고 사용자에게 확인을 요청하십시오:

* 수정을 통해 **해결된 에러보다 새로 발생한 에러가 더 많을 때**
* **동일한 에러가 3번의 시도 후에도 해결되지 않을 때** (심층적인 문제일 가능성)
* 수정에 **아키텍처 변경**이 필요할 때 (단순 빌드 수정을 넘어설 때)
* 에러 원인이 **의존성 누락**일 때 (`npm install` 등 패키지 설치 필요)

## 5단계: 요약 보고

최종 결과를 표시합니다:

* 수정된 에러 목록 (파일 경로 포함)
* 남은 에러 (있는 경우)
* 새로 발생한 에러 (0개여야 함)
* 해결되지 않은 이슈에 대한 후속 권장 조치

## 해결 전략

| 상황 | 조치 |
|-----------|--------|
| 모듈/가져오기 누락 | 패키지 설치 여부 확인 및 설치 명령어 제안 |
| 타입 불일치 | 두 타입의 정의를 모두 읽고 더 좁은 타입으로 수정 |
| 순환 의존성 | 가져오기 그래프로 순환 구조 식별 및 모듈 추출 제안 |
| 버전 충돌 | `package.json` 등 설정 파일의 버전 제약 조건 확인 |
| 빌드 도구 설정 오류 | 설정 파일을 읽고 유효한 기본 설정과 비교 |

**핵심**: 안전을 위해 한 번에 하나의 에러만 수정하십시오. 거창한 리팩토링보다는 문제를 해결하는 최소한의 변경 방식을 우선하십시오.
