# 개발 워크플로우 (Development Workflow)

> 이 문서는 [common/git-workflow.md](git-workflow.md)의 내용을 확장하여, git 작업 이전에 발생하는 전체 기능 개발 프로세스를 다룹니다.

기능 구현 워크플로우는 연구, 계획, TDD, 코드 리뷰, 그리고 git 커밋으로 이어지는 개발 파이프라인을 설명합니다.

## 기능 구현 워크플로우

0. **연구 및 재사용 (Research & Reuse)** *(새로운 구현 시작 전 필수 단계)*
   * **GitHub 코드 검색 우선:** 새로운 코드를 작성하기 전에 `gh search repos` 및 `gh search code`를 실행하여 기존 구현 사례, 템플릿 및 패턴을 찾으십시오.
   * **Exa MCP를 통한 연구:** 계획 단계에서 `exa-web-search` MCP를 사용하여 더 광범위한 연구, 데이터 수집 및 선행 기술 조사를 수행하십시오.
   * **패키지 레지스트리 확인:** 유틸리티 코드를 작성하기 전에 npm, PyPI, crates.io 등의 레지스트리를 검색하십시오. 직접 개발하는 것보다 실전 검증된 라이브러리를 사용하는 것이 우선입니다.
   * **활용 가능한 구현체 검색:** 문제의 80% 이상을 해결할 수 있고 포크(Fork), 포팅(Porting) 또는 래핑(Wrapping)이 가능한 오픈 소스 프로젝트를 찾으십시오.
   * 요구 사항을 충족한다면, 처음부터 새로 시작하기보다 검증된 접근 방식을 채택하거나 이식하는 것을 우선시하십시오.

1. **계획 우선 (Plan First)**
   * **planner** 에이전트를 사용하여 구현 계획을 수립합니다.
   * 코딩 시작 전 기획 문서(PRD), 아키텍처, 시스템 설계, 기술 문서, 작업 목록 등을 생성합니다.
   * 의존성 및 리스크를 식별합니다.
   * 작업을 여러 단계(Phases)로 분해합니다.

2. **TDD 접근 방식**
   * **tdd-guide** 에이전트를 활용합니다.
   * 테스트를 먼저 작성합니다 (RED).
   * 테스트를 통과하도록 기능을 구현합니다 (GREEN).
   * 리팩토링을 수행합니다 (IMPROVE).
   * 80% 이상의 테스트 커버리지를 확보합니다.

3. **코드 리뷰 (Code Review)**
   * 코드 작성 직후 **code-reviewer** 에이전트를 사용합니다.
   * 치명적(CRITICAL) 및 높음(HIGH) 등급의 이슈를 처리합니다.
   * 가능한 경우 중간(MEDIUM) 등급의 이슈도 수정합니다.

4. **커밋 및 푸시 (Commit & Push)**
   * 상세한 커밋 메시지를 작성합니다.
   * 커스텀 커밋(Conventional Commits) 형식을 준수합니다.
   * 커밋 메시지 형식 및 PR 프로세스에 관한 자세한 내용은 [git-workflow.md](git-workflow.md)를 참조하십시오.
